Blockstudio has grown well beyond "a few blocks in a folder". Between
file-based pages, reusable fields, block extensions, and database schemas,
there are now multiple systems that can be registered and discovered through
the filesystem.

That flexibility is powerful, but once a project gets larger it also raises a
simple question:

What exactly is registered right now?

Blockstudio 7.2 adds a new admin overview page to answer that.

## One place for everything

The new screen lives under **Tools > Blockstudio** and gives you a single place
to inspect the things Blockstudio knows about.

Right now it includes dedicated tabs for:

- Blocks
- Extensions
- Fields
- Pages
- Schemas

Instead of rendering these as cards or custom panels, the page uses native
WordPress DataViews tables. That means the UI feels familiar inside wp-admin
and scales better once you have dozens or hundreds of registrations.

## Why this matters

On small projects, it is easy to keep the mental model in your head. On larger
themes, plugins, or multi-instance setups, that breaks down quickly.

You might want to answer questions like:

- Which blocks are actually registered?
- Which extensions target `core/*` and which target specific blocks?
- Which pages are synced and which are disabled?
- Which custom fields are available for reuse?
- Which database schemas exist and what storage backend do they use?

Until now, the answer usually meant opening folders, checking JSON files, or
dropping into PHP to inspect the runtime state.

The new overview turns that into a first-class admin surface.

## Focused on schema-backed registrations

The page is intentionally focused on the parts of Blockstudio that have their
own JSON definition and behave like registries in the system.

That is the reason the first version centers on blocks, extensions, fields,
pages, and schemas. These are the things that define the structure of a
Blockstudio project and are most useful to inspect as data.

Each tab is designed to answer different questions:

- **Blocks** show the registered block types, render mode, API version, and
  attribute count.
- **Extensions** show which blocks they target, along with priority and render
  details.
- **Fields** give a quick inventory of reusable custom field definitions.
- **Pages** show synced file-based pages with post type, status, template
  engine, and sync state.
- **Schemas** surface registered storage schemas with backend, field count, and
  capability scope.

## Built to stay out of the way

Not every install needs another admin screen, so the page can be disabled with
a filter:

```php
add_filter('blockstudio/admin/enabled', '__return_false');
```

That keeps the feature opt-out for teams that want a completely minimal admin
surface.

## Also in 7.2

Another small but useful developer-facing addition in 7.2 is a global SCSS
prelude hook.

That means shared Sass functions, variables, and mixins can now be prepended
before each SCSS file is compiled. In practice, this makes it much easier to
work with Sass toolkits such as Bootstrap without repeating imports in every
block stylesheet.

Combined with custom SCSS import paths, it gives projects a cleaner way to
share styling primitives across a Blockstudio codebase.

## This is the first pass

The current version is intentionally simple. It is an overview page, not a new
settings area.

The goal of the first release is to make the runtime state of a Blockstudio
project visible without digging through code, JSON, or PHP internals. Once
that foundation is in place, the page can grow into better debugging and
inspection workflows over time.

For now, the main win is straightforward:

If Blockstudio registers it, you can see it.
