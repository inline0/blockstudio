Blockstudio 7.1 is the first feature release since the v7 rewrite. It turns
blocks into full-stack applications with RPC, a database layer, scheduled tasks,
components, and more.

## Components

Components are Blockstudio blocks that go through the full pipeline (attributes,
templates, assets, RPC, Tailwind) but never appear in the WordPress editor
inserter. Set `"component": true` in your block.json and the block becomes
available exclusively for programmatic rendering.

```json title="block.json"
{
  "name": "my-theme/card",
  "title": "Card",
  "blockstudio": {
    "component": true,
    "attributes": [
      { "id": "title", "type": "text" },
      { "id": "description", "type": "textarea" }
    ]
  }
}
```

Render components using the string renderer or `bs_render_block()`:

```html
<bs:my-theme-card title="My Card" description="Card content." />
```

```php
bs_render_block([
  'name' => 'my-theme/card',
  'data' => ['title' => 'My Card', 'description' => 'Card content.'],
]);
```

This is useful for building shadcn-style application UI using Blockstudio's
tooling without cluttering the editor inserter.

## HTML tags

Embed Blockstudio blocks anywhere in your HTML output using `<bs:>` tags.
Blockstudio hooks into the output buffer and replaces each tag with the full
block output.

The tag name uses the block's full name with the first hyphen replacing the
namespace separator. For example, `acme/hero` becomes `bs:acme-hero`.

```html
<bs:acme-marketing-cta title="Get started" variant="primary" />
```

It works everywhere: post content, widget areas, theme templates, headers,
footers. Anywhere the tag appears in the final HTML, it gets replaced.

### Syntax

Self-closing tags:

```html
<bs:acme-hero title="Welcome" background="dark" />
```

Paired tags when you need inner content:

```html
<bs:acme-section layout="wide">
  <p>Content inside the block.</p>
</bs:acme-section>
```

Nested tags work too. Inner blocks render first, then the outer block:

```html
<bs:acme-section layout="wide">
  <bs:acme-hero title="Welcome" />
  <bs:acme-card title="Features" />
</bs:acme-section>
```

### Activation

The string renderer is opt-in. Enable it in your `blockstudio.json`:

```json
{
  "stringRenderer": {
    "enabled": true
  }
}
```

Or via the PHP filter:

```php
add_filter('blockstudio/settings/string_renderer/enabled', '__return_true');
```

### Programmatic rendering

You can also apply the string renderer to any string from PHP:

```php
$html = apply_filters('blockstudio/string_renderer/render', $content);
```

## New field types

### Block

Reference another Blockstudio block as an attribute. The referenced block's
fields expand inline using `idStructure` and `overrides`. In templates, the
value returns the rendered block output.

```json title="block.json"
{
  "blockstudio": {
    "attributes": [
      {
        "id": "card",
        "type": "block",
        "block": "mytheme/card",
        "idStructure": "card_{id}"
      }
    ]
  }
}
```

```twig title="index.twig"
<div class="hero">
  <h1>{{ a.title }}</h1>
  {{ a.card }}
</div>
```

### HTML tag

Select an HTML element for semantic flexibility. Presets for headings, text,
structural elements, or all. Exclude specific tags from any preset.

```json
{
  "id": "tag",
  "type": "html-tag",
  "label": "Heading Level",
  "default": "h2",
  "tags": "heading",
  "exclude": ["h5", "h6"]
}
```

```twig
<{{ a.tag }}>{{ a.title }}</{{ a.tag }}>
```

## Database

Blocks can now define a data model via `db.php`. Blockstudio generates CRUD
REST endpoints, validates input against the schema, and provides both a
`bs.db()` JS client and a `Db` PHP API.

```php title="db.php"
return [
    'subscribers' => [
        'storage'    => 'table',
        'capability' => [
            'create' => true,
            'read'   => true,
            'update' => 'edit_posts',
            'delete' => 'manage_options',
        ],
        'fields' => [
            'email' => [
                'type'     => 'string',
                'format'   => 'email',
                'required' => true,
                'validate' => function ($value) {
                    if (str_ends_with($value, '@spam.com')) {
                        return 'This domain is blocked.';
                    }
                    return true;
                },
            ],
            'name' => ['type' => 'string', 'maxLength' => 100],
            'plan' => ['type' => 'string', 'enum' => ['free', 'pro']],
        ],
    ],
];
```

### Four storage modes

- **`table`**: Custom MySQL table, managed automatically via `dbDelta()`.
- **`sqlite`**: Single-file SQLite database with WAL mode. Real SQL queries,
  portable, handles thousands of records.
- **`jsonc`**: One JSON object per line in a flat file. Version-controllable.
- **`meta`**: JSON array in post meta, tied to a specific post.

### JavaScript client

```js
const db = bs.db('my-theme/block', 'subscribers');
const record = await db.create({ email: 'a@b.com', plan: 'pro' });
const all = await db.list({ plan: 'pro' });
await db.update(record.id, { plan: 'free' });
await db.delete(record.id);
```

### PHP API

```php
$db = Db::get('my-theme/block', 'subscribers');
$db->create(['email' => 'a@b.com', 'plan' => 'pro']);
$db->list(['plan' => 'pro']);
$db->update(1, ['plan' => 'free']);
$db->delete(1);
```

### Per-field validation errors

Validation errors return per-field error arrays, making it straightforward to
map errors to form fields:

```json
{
  "code": "blockstudio_db_validation",
  "message": "Validation failed.",
  "data": {
    "status": 400,
    "errors": {
      "email": ["Must be a valid email address.", "This domain is blocked."],
      "plan": ["Must be one of: free, pro."]
    }
  }
}
```

### User-scoped data

Set `userScoped: true` to automatically isolate data per user. Blockstudio
adds a `user_id` column, sets it on create, and filters all operations to the
current user. No manual filtering, no ownership checks.

```php title="db.php"
return [
    'storage'    => 'sqlite',
    'userScoped' => true,
    'fields'     => [
        'text' => ['type' => 'string', 'required' => true],
        'done' => ['type' => 'boolean', 'default' => false],
    ],
];
```

### CSRF protection

Public endpoints are protected by a `X-BS-Token` header, generated per page
load and injected into the JS clients automatically. This prevents cross-site
request forgery even for logged-out users. Use `'open'` instead of `true` when
you need truly unauthenticated access (webhooks, external APIs).

### Form rendering

Fields can define a `component` key referencing a Blockstudio block. Call
`bs_db_form()` to render all fields using their component blocks:

```php title="db.php"
'email' => [
    'type'      => 'string',
    'format'    => 'email',
    'component' => [
        'name' => 'my-theme/field',
        'attributes' => ['label' => 'Email'],
        'innerBlocks' => [
            ['name' => 'my-theme/input', 'attributes' => ['type' => 'email']],
        ],
    ],
],
```

```php title="index.php"
<form>
  <?php bs_db_form('my-theme/app', 'subscribers'); ?>
</form>
```

### Portable mini-apps

Combined with RPC and Cron, blocks can now function as self-contained mini
applications with custom logic, persistent storage, scheduled tasks, and access
control. A block collection with `rpc.php`, `db.php`, `cron.php`, templates, and
assets is a complete full-stack app inside a single folder.

When using JSONC storage, the data lives alongside the code. Copy the block
folder to another site and everything travels with it: templates, styles,
server functions, scheduled tasks, and the data itself.

## Cron

Blocks can define scheduled tasks via `cron.php`. Blockstudio registers them
with WordPress Cron and cleans them up on plugin deactivation.

```php title="cron.php"
return [
    'cleanup' => [
        'schedule' => 'daily',
        'callback' => function () {
            $db = Db::get('my-theme/app', 'logs');
            // delete old entries
        },
    ],
    'sync' => [
        'schedule' => 'hourly',
        'callback' => function () {
            // sync with external API
        },
    ],
];
```

## CLI

The entire Blockstudio feature set is accessible from the command line via
WP-CLI under the `wp bs` prefix:

```bash
wp bs blocks list                    # list all blocks
wp bs blocks list --components       # components only
wp bs db schemas                     # list all database schemas
wp bs db list my-theme/app subs      # query database records
wp bs db create my-theme/app subs --email=a@b.com
wp bs rpc list                       # list all RPC functions
wp bs rpc call my-theme/app subscribe --email=a@b.com
wp bs cron list                      # list scheduled jobs
wp bs cron run my-theme/app cleanup  # run a job manually
wp bs tailwind compile --file=t.html # compile Tailwind CSS
wp bs tailwind cache clear           # clear Tailwind cache
wp bs scss compile --file=style.scss # compile SCSS to CSS
wp bs fields list                    # list custom field definitions
wp bs assets list --type=global      # list assets by type
wp bs settings get tailwind/enabled  # read settings
```

Every command supports `--format=json` for scripting and piping.

## RPC

Blocks can now define PHP functions that the frontend can call. Drop a
`rpc.php` into any block directory and Blockstudio handles the rest:
REST endpoint routing, authentication, and a tiny JS client.

```php title="rpc.php"
return [
    'subscribe' => function (array $params): array {
        $email = sanitize_email($params['email']);
        return ['success' => true];
    },
];
```

```js
const result = await bs.fn('subscribe', { email: 'user@example.com' });
```

This pairs naturally with the WordPress Interactivity API. Your store actions can
call server functions and update reactive state with the response.

Functions are authenticated by default. Access control options include
`public` for unauthenticated access, `capability` for WordPress capability
checks (string or array), and `methods` to control allowed HTTP methods.
Cross-block calling works too: any block can call functions on any other block
by passing the block name as the third argument.

## RichText and MediaPlaceholder in repeaters

The `RichText` and `MediaPlaceholder` template components now work inside repeater
fields. Use bracket notation in the `attribute` prop to target the correct row.

```twig
{% for item in a.items %}
  <div class="card">
    <MediaPlaceholder attribute="items[{{ loop.index0 }}].image" />
    <RichText tag="h2" attribute="items[{{ loop.index0 }}].heading" />
    <RichText tag="p" attribute="items[{{ loop.index0 }}].content" />
  </div>
{% endfor %}
```

This was a frequently requested feature. Editable headings, paragraphs, and media
drop zones now work per-row in any repeater field.

## Custom field conditions with idStructure

Conditions inside custom field definitions now automatically rewrite their IDs
when `idStructure` is applied. A toggle that controls sibling fields keeps
working after expansion. Reference-level conditions can also be applied to hide
an entire field group behind an external toggle.

```json
// Field definition: conditions use original IDs
{ "id": "min", "conditions": [[{ "id": "enable", ... }]] }

// Usage: idStructure remaps both field IDs and condition IDs
{ "type": "custom/font-size", "idStructure": "title_font_{id}" }
// → title_font_min condition now references title_font_enable
```

## SCSS code fields

Code fields now support `language: "scss"`. When set, the field value is compiled
through ScssPhp before rendering as CSS, both in the editor and on the frontend.

```json
{
  "type": "code",
  "id": "styles",
  "label": "Custom Styles",
  "language": "scss",
  "asset": true
}
```

Variables, nesting, mixins, and all standard SCSS features work out of the box.
In the editor, compilation happens server-side with debounced requests so you see
results as you type.

```scss
%selector% {
  $primary: #3b82f6;

  h1 {
    color: $primary;
  }

  &:hover {
    background: lighten($primary, 40%);
  }
}
```

No additional settings required. Just set the language to `scss` and it works.

## Standard WordPress block support

Blockstudio now auto-registers standard WordPress blocks found in your
Blockstudio directories. Drop a `@wordpress/create-block` build output into your
`blockstudio/` folder and it gets registered automatically, no separate plugin
registration code needed. Blockstudio and standard WordPress blocks can live
side by side in the same directory structure.
