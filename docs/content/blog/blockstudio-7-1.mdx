Blockstudio 7.1 is the first feature release since the v7 rewrite. It introduces
the block field type, components, the string renderer, SCSS support for code
fields, and lays the groundwork for more features to come.

## Block field

The new `block` field type lets you reference another Blockstudio block as an
attribute. The referenced block's fields expand inline using the same
`idStructure` and `overrides` system as custom fields. In templates, the value
returns the rendered block output.

```json title="block.json"
{
  "blockstudio": {
    "attributes": [
      {
        "id": "card",
        "type": "block",
        "block": "mytheme/card",
        "idStructure": "card_{id}"
      }
    ]
  }
}
```

```twig title="index.twig"
<div class="hero">
  <h1>{{ a.title }}</h1>
  {{ a.card }}
</div>
```

This works with both regular blocks and component blocks. It uses the same
expansion logic as custom fields, so `idStructure` and `overrides` work
identically. The `returnFormat` option controls whether you get rendered HTML,
raw data, or both.

## Components

Components are Blockstudio blocks that go through the full pipeline (attributes,
templates, assets, RPC, Tailwind) but never appear in the WordPress editor
inserter. Set `"component": true` in your block.json and the block becomes
available exclusively for programmatic rendering.

```json title="block.json"
{
  "name": "my-theme/card",
  "title": "Card",
  "blockstudio": {
    "component": true,
    "attributes": [
      { "id": "title", "type": "text" },
      { "id": "description", "type": "textarea" }
    ]
  }
}
```

Render components using the string renderer or `bs_render_block()`:

```html
<bs:card title="My Card" description="Card content." />
```

```php
bs_render_block([
  'name' => 'my-theme/card',
  'data' => ['title' => 'My Card', 'description' => 'Card content.'],
]);
```

This is useful for building shadcn-style application UI using Blockstudio's
tooling without cluttering the editor inserter.

## String renderer

The string renderer lets you embed Blockstudio blocks anywhere in your HTML
output using simple tags. Write `<bs:block-name>` anywhere on the page and
Blockstudio renders the full block output in its place.

```html
<bs:marketing-cta title="Get started" variant="primary" />
```

It hooks into the output buffer, so it works everywhere: post content, widget
areas, theme templates, headers, footers. Anywhere the tag appears in the final
HTML, it gets replaced.

Element attributes map directly to block attributes, so passing data works the
same as it does in the editor.

### Syntax

Self-closing tags for simple blocks:

```html
<bs:hero title="Welcome" background="dark" />
```

Paired tags when you need inner content:

```html
<bs:section layout="wide">
  <p>Content inside the block.</p>
</bs:section>
```

Nested tags work too. Inner blocks render first, then the outer block:

```html
<bs:section layout="wide">
  <bs:hero title="Welcome" />
  <bs:card title="Features" />
</bs:section>
```

Namespace prefix with double-dash for disambiguation:

```html
<bs:acme--hero title="Branded" />
```

Without a namespace, Blockstudio matches the first registered block with that
slug. With a namespace (`acme--hero`), it resolves to exactly `acme/hero`.

### Activation

The string renderer is opt-in. Enable it in your `blockstudio.json`:

```json
{
  "stringRenderer": {
    "enabled": true
  }
}
```

Or via the PHP filter:

```php
add_filter('blockstudio/settings/string_renderer/enabled', '__return_true');
```

### Programmatic rendering

You can also apply the string renderer to any string from PHP:

```php
$html = apply_filters('blockstudio/string_renderer/render', $content);
```

## SCSS code fields

Code fields now support `language: "scss"`. When set, the field value is compiled
through ScssPhp before rendering as CSS, both in the editor and on the frontend.

```json
{
  "type": "code",
  "id": "styles",
  "label": "Custom Styles",
  "language": "scss",
  "asset": true
}
```

Variables, nesting, mixins, and all standard SCSS features work out of the box.
In the editor, compilation happens server-side with debounced requests so you see
results as you type.

```scss
%selector% {
  $primary: #3b82f6;

  h1 {
    color: $primary;
  }

  &:hover {
    background: lighten($primary, 40%);
  }
}
```

No additional settings required. Just set the language to `scss` and it works.

## RichText and MediaPlaceholder in repeaters

The `RichText` and `MediaPlaceholder` template components now work inside repeater
fields. Use bracket notation in the `attribute` prop to target the correct row.

```twig
{% for item in a.items %}
  <div class="card">
    <MediaPlaceholder attribute="items[{{ loop.index0 }}].image" />
    <RichText tag="h2" attribute="items[{{ loop.index0 }}].heading" />
    <RichText tag="p" attribute="items[{{ loop.index0 }}].content" />
  </div>
{% endfor %}
```

This was a frequently requested feature. Editable headings, paragraphs, and media
drop zones now work per-row in any repeater field.

## Server functions

Blocks can now define PHP functions that the frontend can call. Drop a
`rpc.php` into any block directory and Blockstudio handles the rest:
REST endpoint routing, authentication, and a tiny JS client.

```php title="rpc.php"
return [
    'subscribe' => function (array $params): array {
        $email = sanitize_email($params['email']);
        return ['success' => true];
    },
];
```

```js
const result = await bs.fn('subscribe', { email: 'user@example.com' });
```

This pairs naturally with the WordPress Interactivity API. Your store actions can
call server functions and update reactive state with the response.

Functions are authenticated by default. Add `'public' => true` to allow
unauthenticated access. Cross-block calling works too: any block can call
functions on any other block by passing the block name as the third argument.
