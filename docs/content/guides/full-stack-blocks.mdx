# Full-Stack Blocks

Blockstudio 7.1 lets you build complete applications inside a block folder.
A single directory can contain the UI, server logic, data model, scheduled tasks,
and even the data itself.

This guide walks through building a per-user todo app as a single block. By the
end you'll have a working application with authentication, CRUD, live UI updates,
and automated cleanup, all inside one folder.

## The file structure

```
blockstudio/
  todo/
    block.json            ← block definition
    index.php             ← template with Interactivity API
    style.css             ← styles
    script-inline.js      ← client logic
    db.php                ← data model
    rpc.php               ← custom operations
    cron.php              ← scheduled cleanup
```

Seven files. No build step, no external dependencies, no separate plugin.

## Step 1: Block definition

```json title="block.json"
{
  "$schema": "https://blockstudio.dev/schema/block",
  "name": "my-theme/todo",
  "title": "Todo App",
  "category": "widgets",
  "icon": "editor-ul",
  "blockstudio": true
}
```

No attributes needed. All state lives in the database and the Interactivity API
context.

## Step 2: Data model

The `db.php` defines the schema. Each todo belongs to a user via `user_id`.
Public read so the block renders for logged-out users (empty state), but writes
require authentication.

```php title="db.php"
<?php
return [
    'storage'    => 'sqlite',
    'userScoped' => true,
    'capability' => [
        'create' => true,
        'read'   => true,
        'update' => true,
        'delete' => true,
    ],
    'fields' => [
        'text' => [
            'type'      => 'string',
            'required'  => true,
            'minLength' => 1,
            'maxLength' => 500,
        ],
        'done' => [
            'type'    => 'boolean',
            'default' => false,
        ],
    ],
];
```

`userScoped: true` automatically adds a `user_id` column, sets it on create,
and filters all queries to the current user. No manual user handling needed.

SQLite storage means the database file lives at `todo/db/default.sqlite`. Copy
the block folder to another site and the data comes with it.

This gives you five REST endpoints automatically:

```
POST   /wp-json/blockstudio/v1/db/my-theme-todo/default
GET    /wp-json/blockstudio/v1/db/my-theme-todo/default
GET    /wp-json/blockstudio/v1/db/my-theme-todo/default/{id}
PUT    /wp-json/blockstudio/v1/db/my-theme-todo/default/{id}
DELETE /wp-json/blockstudio/v1/db/my-theme-todo/default/{id}
```

## Step 3: Custom server logic

Since `userScoped: true` auto-filters to the current user, we don't need
a custom `my_todos` endpoint. The built-in `bs.db().list()` already returns
only the current user's records. RPC is only needed for operations beyond CRUD:
toggling and bulk clearing.

```php title="rpc.php"
<?php
use Blockstudio\Db;

return [
    'toggle' => [
        'callback' => function (array $params): array {
            $db   = Db::get('my-theme/todo');
            $id   = (int) ($params['id'] ?? 0);
            $todo = $db->get_record($id);

            if (!$todo) {
                return ['error' => 'Not found.'];
            }

            // userScoped already ensures this is the current user's record.
            $db->update($id, ['done' => !$todo['done']]);

            return ['success' => true];
        },
        'public' => false,
    ],
    'clear_done' => [
        'callback' => function (array $params): array {
            $db    = Db::get('my-theme/todo');
            $todos = $db->list(); // userScoped: only current user's todos.
            $cleared = 0;

            foreach ($todos as $todo) {
                if ($todo['done']) {
                    $db->delete((int) $todo['id']);
                    ++$cleared;
                }
            }

            return ['cleared' => $cleared];
        },
        'public' => false,
    ],
];
```

## Step 4: Scheduled cleanup

Old completed todos get cleaned up automatically. Cron runs daily.

```php title="cron.php"
<?php
use Blockstudio\Db;

return [
    'cleanup_old_todos' => [
        'schedule' => 'daily',
        'callback' => function () {
            $db    = Db::get('my-theme/todo');
            $todos = $db->list();

            foreach ($todos as $todo) {
                $is_done = $todo['done'];
                $is_old  = strtotime($todo['updated_at']) < strtotime('-30 days');

                if ($is_done && $is_old) {
                    $db->delete((int) $todo['id']);
                }
            }
        },
    ],
];
```

## Step 5: The template

The template uses the WordPress Interactivity API for reactive state. PHP loads
the initial todos server-side, then JavaScript handles interactions.

```php title="index.php"
<?php
$user_id = get_current_user_id();
$db      = \Blockstudio\Db::get('my-theme/todo');
$todos   = ($user_id && $db) ? $db->list() : []; // userScoped: auto-filters to current user.

$context = wp_json_encode([
    'todos'   => $todos,
    'newText' => '',
    'loading' => false,
    'userId'  => $user_id,
]);
?>

<div
    data-wp-interactive="myTheme/todo"
    data-wp-context='<?php echo esc_attr($context); ?>'
    useBlockProps
    class="todo-app"
>
    <?php if (!$user_id) : ?>
        <p class="todo-login">Log in to manage your todos.</p>
    <?php else : ?>
        <div class="todo-input">
            <input
                type="text"
                placeholder="What needs to be done?"
                data-wp-bind--value="context.newText"
                data-wp-on--input="actions.updateText"
                data-wp-on--keydown="actions.handleKeydown"
            />
            <button
                data-wp-on--click="actions.addTodo"
                data-wp-bind--disabled="context.loading"
            >Add</button>
        </div>

        <ul class="todo-list" data-wp-each="context.todos" data-wp-each-key="id">
            <template>
                <li>
                    <label>
                        <input
                            type="checkbox"
                            data-wp-bind--checked="context.item.done"
                            data-wp-on--change="actions.toggleTodo"
                        />
                        <span
                            data-wp-text="context.item.text"
                            data-wp-class--done="context.item.done"
                        ></span>
                    </label>
                    <button
                        class="todo-delete"
                        data-wp-on--click="actions.deleteTodo"
                    >&times;</button>
                </li>
            </template>
        </ul>

        <div class="todo-footer">
            <span data-wp-text="state.remainingText"></span>
            <button
                data-wp-on--click="actions.clearDone"
                data-wp-bind--hidden="!state.hasDone"
            >Clear completed</button>
        </div>
    <?php endif; ?>
</div>
```

## Step 6: Client logic

The inline script auto-detects the block name from its `data-block` attribute,
so `bs.fn()` and `bs.db()` work without passing it explicitly.

```js title="script-inline.js"
import { store, getContext } from '@wordpress/interactivity';

store('myTheme/todo', {
    state: {
        get remainingText() {
            const ctx = getContext();
            const remaining = ctx.todos.filter(t => !t.done).length;
            return `${remaining} item${remaining !== 1 ? 's' : ''} left`;
        },
        get hasDone() {
            const ctx = getContext();
            return ctx.todos.some(t => t.done);
        },
    },
    actions: {
        updateText(event) {
            const ctx = getContext();
            ctx.newText = event.target.value;
        },
        handleKeydown(event) {
            if (event.key === 'Enter') {
                const { actions } = store('myTheme/todo');
                actions.addTodo();
            }
        },
        async addTodo() {
            const ctx = getContext();
            const text = ctx.newText.trim();
            if (!text) return;

            ctx.loading = true;
            const db = bs.db('my-theme/todo');
            const record = await db.create({ text, done: false });
            // user_id is auto-set by userScoped.

            ctx.todos = [record, ...ctx.todos];
            ctx.newText = '';
            ctx.loading = false;
        },
        async toggleTodo() {
            const ctx = getContext();
            const todo = ctx.item;
            await bs.fn('toggle', { id: todo.id });
            todo.done = !todo.done;
        },
        async deleteTodo() {
            const ctx = getContext();
            const todo = ctx.item;
            const db = bs.db('my-theme/todo');
            await db.delete(todo.id);
            const parent = getContext({ level: 1 });
            parent.todos = parent.todos.filter(t => t.id !== todo.id);
        },
        async clearDone() {
            const ctx = getContext();
            await bs.fn('clear_done', {});
            ctx.todos = ctx.todos.filter(t => !t.done);
        },
    },
});
```

## Managing from the CLI

```bash
# List all todos
wp bs db list my-theme/todo

# Create a todo for user 1
wp bs db create my-theme/todo --text="Buy milk" --user_id=1

# Run the cleanup cron manually
wp bs cron run my-theme/todo cleanup_old_todos

# Call the toggle RPC
wp bs rpc call my-theme/todo toggle --id=1
```

## What you get

From seven files in a single folder:

| Capability | Source |
|-----------|--------|
| Block UI with reactive state | `index.php` + `script-inline.js` |
| Scoped styles | `style.css` |
| Per-user data with SQLite | `db.php` |
| Auto-generated CRUD endpoints | `db.php` |
| Input validation | `db.php` field rules |
| Custom server operations | `rpc.php` |
| Automated cleanup | `cron.php` |
| CLI management | `wp bs db/rpc/cron` |
| Portability | Copy the folder |

No webpack, no separate database migrations, no REST controller classes, no
plugin boilerplate. Drop the folder into any theme's `blockstudio/` directory
and you have a working todo app.

# Example 2: Newsletter Signup

Not every app needs per-user data. A newsletter signup block is public: anyone
can subscribe, only admins can view and manage subscribers. No `userScoped`,
no authentication required for the main action.

## File structure

```
blockstudio/
  newsletter/
    block.json
    index.php
    style.css
    script-inline.js
    db.php
    rpc.php
    cron.php
```

## Data model

Public create (anyone can subscribe), admin-only read/delete. Custom validation
blocks disposable emails.

```php title="db.php"
<?php
return [
    'storage'    => 'table',
    'capability' => [
        'create' => true,
        'read'   => 'manage_options',
        'update' => 'manage_options',
        'delete' => 'manage_options',
    ],
    'fields' => [
        'email' => [
            'type'     => 'string',
            'format'   => 'email',
            'required' => true,
            'validate' => function ($value) {
                $blocked = ['mailinator.com', 'tempmail.com', 'throwaway.email'];
                $domain  = substr($value, strpos($value, '@') + 1);
                if (in_array($domain, $blocked, true)) {
                    return 'Please use a permanent email address.';
                }
                return true;
            },
        ],
        'name' => [
            'type'      => 'string',
            'maxLength' => 100,
        ],
        'status' => [
            'type'    => 'string',
            'enum'    => ['active', 'unsubscribed'],
            'default' => 'active',
        ],
    ],
    'hooks' => [
        'after_create' => function ($params) {
            wp_mail(
                $params['record']['email'],
                'Welcome!',
                'Thanks for subscribing' . ($params['record']['name'] ? ', ' . $params['record']['name'] : '') . '.'
            );
        },
    ],
];
```

## Custom logic

An unsubscribe endpoint that's public (works from email links without login):

```php title="rpc.php"
<?php
use Blockstudio\Db;

return [
    'unsubscribe' => [
        'callback' => function (array $params): array {
            $email = sanitize_email($params['email'] ?? '');
            $db    = Db::get('my-theme/newsletter');
            $subs  = $db->list(['email' => $email]);

            if (empty($subs)) {
                return ['error' => 'Email not found.'];
            }

            $db->update((int) $subs[0]['id'], ['status' => 'unsubscribed']);

            return ['success' => true, 'message' => 'You have been unsubscribed.'];
        },
        'public' => true,
    ],
    'stats' => [
        'callback'   => function (array $params): array {
            $db = Db::get('my-theme/newsletter');
            return [
                'total'  => count($db->list()),
                'active' => count($db->list(['status' => 'active'])),
            ];
        },
        'capability' => 'manage_options',
    ],
];
```

## Scheduled cleanup

Remove unsubscribed emails after 90 days:

```php title="cron.php"
<?php
use Blockstudio\Db;

return [
    'cleanup' => [
        'schedule' => 'weekly',
        'callback' => function () {
            $db    = Db::get('my-theme/newsletter');
            $unsub = $db->list(['status' => 'unsubscribed']);

            foreach ($unsub as $row) {
                if (strtotime($row['updated_at']) < strtotime('-90 days')) {
                    $db->delete((int) $row['id']);
                }
            }
        },
    ],
];
```

## Client logic

```js title="script-inline.js"
import { store, getContext } from '@wordpress/interactivity';

store('myTheme/newsletter', {
    actions: {
        setName(event) {
            getContext().name = event.target.value;
        },
        setEmail(event) {
            getContext().email = event.target.value;
        },
        handleKeydown(event) {
            if (event.key === 'Enter') {
                store('myTheme/newsletter').actions.subscribe();
            }
        },
        async subscribe() {
            const ctx = getContext();
            const email = ctx.email.trim();
            if (!email) return;

            ctx.loading = true;
            ctx.error = '';

            const db = bs.db('my-theme/newsletter');
            const result = await db.create({
                email,
                name: ctx.name.trim(),
            });

            if (result.data?.errors) {
                const first = Object.values(result.data.errors)[0];
                ctx.error = first[0];
                ctx.loading = false;
                return;
            }

            if (result.code) {
                ctx.error = result.message || 'Something went wrong.';
                ctx.loading = false;
                return;
            }

            ctx.success = true;
            ctx.loading = false;
        },
    },
});
```

## Comparison

| | Todo App | Newsletter |
|---|---------|-----------|
| **Auth model** | `userScoped: true` | Public create, admin read |
| **Storage** | SQLite (portable) | MySQL table (production) |
| **User filtering** | Automatic | Not needed |
| **Custom RPC** | Toggle, clear done | Unsubscribe, stats |
| **Cron** | Delete old completed | Remove unsubscribed |
| **Validation** | Length constraints | Email format + domain block |

Same pattern, different access models. The building blocks are identical:

- `db.php` for the data model
- `rpc.php` for business logic
- `cron.php` for background tasks
- `index.php` + `script-inline.js` for the UI
- `style.css` for the look

Each block is a self-contained application. Compose multiple blocks on a page
for a full dashboard. Use [components](/docs/blocks/components) for shared UI
pieces that don't appear in the editor. Use the
[block field](/docs/blocks/attributes/block-field) to embed one block's output
inside another.
