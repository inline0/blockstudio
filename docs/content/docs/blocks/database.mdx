# Database

Blocks can define a data model via a `db.php` file. Blockstudio generates CRUD
REST endpoints, validates input, manages storage, and provides a `bs.db()` JS
client.

## Setup

Create a `db.php` in your block directory that returns a schema definition:

```
blocks/my-block/
├── block.json
├── index.twig
├── rpc.php     ← custom logic
├── db.php      ← data model
└── style.css
```

### Single schema

```php title="db.php"
return [
    'storage'    => 'table',
    'capability' => [
        'create' => true,
        'read'   => true,
        'update' => 'edit_posts',
        'delete' => 'manage_options',
    ],
    'fields' => [
        'email' => ['type' => 'string', 'format' => 'email', 'required' => true],
        'name'  => ['type' => 'string', 'maxLength' => 100],
        'plan'  => ['type' => 'string', 'enum' => ['free', 'pro'], 'default' => 'free'],
    ],
];
```

### Multiple schemas

Return a named array to define multiple schemas per block:

```php title="db.php"
return [
    'subscribers' => [
        'storage' => 'table',
        'fields'  => [
            'email' => ['type' => 'string', 'format' => 'email', 'required' => true],
            'plan'  => ['type' => 'string', 'enum' => ['free', 'pro']],
        ],
    ],
    'logs' => [
        'storage' => 'jsonc',
        'fields'  => [
            'action'  => ['type' => 'string', 'required' => true],
            'details' => ['type' => 'string'],
        ],
    ],
];
```

## Storage Modes

### Custom Table (`storage: "table"`)

Each record is a database row. Blockstudio creates and manages the table
automatically via `dbDelta()`. Adding fields to the schema adds columns
on the next load.

### JSONC Files (`storage: "jsonc"`)

Records are stored as one JSON object per line in a `.jsonc` file inside a `db/`
subfolder of the block directory. File-based, portable, and version-controllable.
Comments (lines starting with `//`) are preserved.

```
blocks/my-block/
├── db.php
└── db/
    └── logs.jsonc    ← auto-created on first write
```

### Post Meta (`storage: "meta"`)

Records are stored as a JSON array in post meta. Requires a `postId` in the
schema to specify which post owns the data.

```php title="db.php"
return [
    'storage' => 'meta',
    'postId'  => 42,
    'fields'  => [ /* ... */ ],
];
```

## JavaScript Client

Blockstudio injects `bs.db()` on pages with database blocks:

```js
const db = bs.db('my-theme/block', 'subscribers');

// Create
const record = await db.create({ email: 'a@b.com', plan: 'pro' });

// List (with optional filters)
const all = await db.list();
const pros = await db.list({ plan: 'pro', limit: 10 });

// Read
const one = await db.get(record.id);

// Update
const updated = await db.update(record.id, { plan: 'enterprise' });

// Delete
await db.delete(record.id);
```

The first argument is the block name and the second is the schema name. For
single-schema blocks, the schema name can be omitted (defaults to `"default"`).

## PHP API

Use the `\Blockstudio\Db` class for server-side access from `rpc.php`, templates,
or any PHP code:

```php
$db = \Blockstudio\Db::get('mytheme/block', 'subscribers');

$record = $db->create(['email' => 'a@b.com', 'plan' => 'pro']);
$all    = $db->list(['plan' => 'pro']);
$one    = $db->get_record(1);
$db->update(1, ['plan' => 'enterprise']);
$db->delete(1);
```

For single-schema blocks, omit the schema name (defaults to `"default"`):

```php
$db = \Blockstudio\Db::get('mytheme/block');
```

## Access Control

Per-operation capability control. Use `true` for public access, a capability
string, or an array of capabilities (user needs any one):

```php title="db.php"
'capability' => [
    'create' => true,                              // public
    'read'   => true,                              // public
    'update' => 'edit_posts',                      // single capability
    'delete' => ['delete_posts', 'manage_options'], // any of these
],
```

When `capability` is omitted for an operation, it defaults to requiring
authentication (logged-in user).

## Field Types

| Type | MySQL Column | Description |
|------|-------------|-------------|
| `string` | `varchar(maxLength)` | Text, default maxLength 255 |
| `integer` | `bigint(20)` | Whole numbers |
| `number` | `decimal(20,6)` | Decimal numbers |
| `boolean` | `tinyint(1)` | True/false |
| `text` | `longtext` | Long text content |

## Validation

All writes are validated against the schema before storage:

| Rule | Description |
|------|-------------|
| `required` | Field must be present on create |
| `type` | Value must match the declared type |
| `enum` | Value must be one of the listed options |
| `maxLength` | String must not exceed this length |
| `minLength` | String must be at least this length |
| `format: "email"` | Must be a valid email address |
| `format: "url"` | Must be a valid URL |
| `validate` | Custom callback for server-side validation |

### Custom Validation

Add a `validate` callback to any field for custom server-side checks:

```php title="db.php"
'email' => [
    'type'     => 'string',
    'format'   => 'email',
    'required' => true,
    'validate' => function ($value, $data) {
        if (str_ends_with($value, '@banned.com')) {
            return 'This domain is not allowed.';
        }
        return true;
    },
],
```

Return `true` to pass, or a string with the error message to fail.

### Error Response

Validation errors return a `400` status with per-field error arrays:

```json
{
  "code": "blockstudio_db_validation",
  "message": "Validation failed.",
  "data": {
    "status": 400,
    "errors": {
      "email": ["Must be a valid email address."],
      "plan": ["Must be one of: free, pro, enterprise."]
    }
  }
}
```

Each field maps to an array of error strings, so multiple validation rules
can fail on the same field. This format makes it straightforward to display
errors next to form fields on the frontend.

## REST Endpoints

All endpoints are under the `blockstudio/v1/db/` namespace:

```
/db/{block-name}/{schema-name}
```

The block name has `/` replaced by `-`. For single-schema blocks, the schema
name is `default`.

| Method | Endpoint | Description |
|--------|----------|-------------|
| `GET` | `/db/{block}/{schema}` | List records |
| `POST` | `/db/{block}/{schema}` | Create a record |
| `GET` | `/db/{block}/{schema}/{id}` | Get a record |
| `PUT` | `/db/{block}/{schema}/{id}` | Update a record |
| `DELETE` | `/db/{block}/{schema}/{id}` | Delete a record |

### Query Parameters (list)

| Parameter | Description |
|-----------|-------------|
| `limit` | Maximum records (default 50, max 100) |
| `offset` | Skip records |
| Any field name | Filter by exact value |

## Filter

Use the `blockstudio/database` filter to modify schemas programmatically:

```php
add_filter('blockstudio/database', function (array $schemas): array {
    $schemas['my-theme/leads:default']['fields']['source'] = [
        'type' => 'string',
        'default' => 'website',
    ];
    return $schemas;
});
```

## Combining with RPC

`db.php` and `rpc.php` are independent. `db.php` generates CRUD endpoints
automatically. `rpc.php` defines custom server functions. Either works on its
own, and they can be combined when you need both standard data operations and
custom logic.

For example, use `db.php` for the data model and `rpc.php` for operations that
go beyond simple CRUD:

```php title="rpc.php"
return [
    'export_csv' => [
        'callback'   => function (array $params): array {
            $db = \Blockstudio\Db::get('my-theme/leads', 'subscribers');
            $rows = $db->list();
            // custom export logic
            return ['count' => count($rows)];
        },
        'capability' => 'manage_options',
    ],
];
```

See the [RPC documentation](/docs/blocks/functions) for details on server
functions.

## Hooks

Actions fire before and after every write operation. All hooks receive a single
`$params` array. See the [PHP Hooks](/docs/blocks/hooks/php#database) reference
for the full list.

### Inline hooks

Define hooks directly in `db.php` using the `hooks` key. They are scoped to
that schema automatically:

```php title="db.php"
return [
    'storage' => 'table',
    'fields'  => [
        'email' => ['type' => 'string', 'format' => 'email', 'required' => true],
        'plan'  => ['type' => 'string', 'enum' => ['free', 'pro']],
    ],
    'hooks' => [
        'after_create' => function ($params) {
            wp_mail($params['record']['email'], 'Welcome!', 'Thanks for subscribing.');
        },
        'before_delete' => function ($params) {
            error_log('Deleting record ' . $params['id']);
        },
    ],
];
```

### Global hooks

Use WordPress actions to listen across all schemas:

```php title="functions.php"
add_action('blockstudio/db/after_create', function ($params) {
    if ('my-theme/app' === $params['block']) {
        // runs for all schemas in this block
    }
});
```

## Portability

With JSONC storage, data lives alongside code. A block folder with `db.php`,
`rpc.php`, templates, styles, and `db/*.jsonc` files is a complete portable
application. Copy it to another site and everything travels together: the UI,
the logic, the access control, and the data itself.

```
my-app/
├── block.json
├── index.twig
├── style.css
├── rpc.php        ← custom endpoints
├── db.php         ← data schema
└── db/
    └── entries.jsonc  ← portable data
```
